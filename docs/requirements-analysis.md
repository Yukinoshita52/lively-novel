# Lively Novel — 需求分析文档

> 日期：2026-06-05
> 版本：v1.1

---

## 1. 项目背景

Lively Novel 是**七牛云 72h 项目开发**比赛的参赛作品，选题为**题目三：AI 小说转剧本工具**。

**题目要求：**
- 能将 3 个章节以上的小说文本自动转换为结构化剧本（YAML 格式）
- 让作者可以快速获得可编辑、可进一步打磨的剧本初稿
- 额外写一篇文档，定义剧本的 YAML Schema，并说明设计原因

**项目名含义：** 小说是一种**静态文本载体**，而剧本是一种**面向表演和视觉呈现的动态载体**。Lively Novel 帮助作者将静态小说转化为能够"动起来"的剧本初稿。

---

## 2. 目标用户

### 2.1 主用户：原创/网文作者

**画像：** 写了小说、梦想 IP 改编（影视/短剧/广播剧/漫画）的人。他们有故事，但**不懂剧本格式**。

**选择理由：**
- 题目原文"降低改编门槛"最贴合这类人群
- 他们是痛点最深的人——有创作力但缺专业知识和转换能力
- 最容易做出有共鸣的产品故事和 demo

### 2.2 次要用户

| 用户群体 | 特征 | 受益方式 |
|---|---|---|
| 新手/改编编剧 | 懂剧本格式，要效率 | 省掉机械拆场、提白、排版 |
| 短剧/广播剧团队 | 要批量、快、标准化 | 批量转换、一致性保证 |
| 影视公司 IP 评估 | 快速判断小说可改编性 | 快速看到"改出来长什么样" |

---

## 3. 用户痛点

按真实发生顺序排列：

| # | 痛点 | 描述 | 严重度 |
|---|---|---|---|
| 1 | 格式是黑箱 | 剧本格式（场景标题、动作行、角色提示、对白、括号提示、转场）对小说作者是天书 | ★★★★ |
| 2 | 对白埋在散文里 | "'你够了。'她别过脸，声音发颤"——要把"谁、说了什么、带着什么情绪"干净地抽出来 | ★★★★ |
| 3 | 内心戏无处安放 | 小说大量靠心理描写和内心独白。镜头拍不出"想法"，必须转成动作/对白/画外音(V.O.)/可视细节。**这是改编的灵魂，也是作者最容易卡死的地方** | ★★★★★ |
| 4 | 不知道在哪"拆场" | 小说一章是连续叙述，剧本要按地点/时间切成一个个场景 | ★★★★ |
| 5 | "讲述"要变"呈现" | 小说能直接写"她很绝望"，剧本得靠看得见的动作演出来 | ★★★★ |
| 6 | 跨章一致性 | 3 章以上，人物名、地点、线索得统一，需要一份全局的人物表/场景表 | ★★★ |
| 7 | 纯体力活劝退 | 就算硬做，机械重排和标注也耗到放弃 | ★★★★ |

---

## 4. 用户需求（Job-to-be-done）

> 「我有几章小说，我想**不用学剧本知识、不做体力活**，就快速拿到一份**格式正确、结构清晰、能看出戏剧雏形**的剧本初稿，然后我自己在上面改。」

**核心矛盾：** 小说是"静态叙述"，剧本是"动态呈现"，中间这层转译，作者缺少剧本创作知识（e.g.剧本格式、将内心戏视觉化）。Lively Novel 要做的就是替他跨过这层。

## 5. 功能需求

### 5.1 必做功能（MVP）

| 功能 | 描述 | 对应痛点 |
|---|---|---|
| 小说导入 | 支持上传 `.txt` 文件，自动识别章节边界 | 7 |
| 剧本类型选择 | 动画/影视/短剧/广播剧/话剧等，影响生成策略 | - |
| 两阶段转换 | 先全局分析（人物/场景/剧情），再逐场生成 YAML 结构 | 1,2,3,4,5,6 |
| 剧本预览 | YAML 结构的渲染视图（标准剧本排版） | 1 |
| 逐场打磨 | 编辑 YAML 字段 / 选择视觉化手法 / AI 重生成 | 3,5 |
| **YAML 导出** | **核心交付**：符合 Schema 的结构化剧本文件 | 题目要求 |
| 可读剧本导出 | YAML 的人类友好渲染（衍生功能） | 7 |
| **Schema 文档** | 定义 YAML Schema 并说明设计原因 | 题目要求 |

### 5.2 加分功能（时间允许）

| 功能 | 描述 | 价值 |
|---|---|---|
| 多格式导入 | `.epub`、`.md`、`.docx`、`.pdf` 支持 | 降低使用门槛 |
| 流式生成展示 | 实时展示 AI 生成过程 | 提升体验感 |
| 全局可视化 | 人物关系图谱、剧情时间线、世界地图 | demo 亮点 |

### 5.3 Agent 化演进需求

Agent 化是比赛 MVP 之后在 `master` 分支上的长期演进目标，不改变题目要求的核心交付：结构化 YAML 剧本与 YAML Schema 文档。

| 需求 | 描述 | 价值 |
|---|---|---|
| Agent 运行轨迹 | 记录 run、step、tool call、guardrail、artifact 与失败原因 | 让长任务可观察、可排错、可恢复 |
| 工具化能力封装 | 将章节读取、切场、单场生成、状态更新、校验、保存、导出包装为受控工具 | 为 planner、多 Agent、MCP 暴露打基础 |
| 受限规划 | 让模型在白名单工具和最大步数内生成结构化计划，失败时回退固定流程 | 从固定 workflow 逐步过渡到可控 Agent |
| 质量护栏 | 对输入、工具调用、输出 YAML、语言漂移和人物一致性做校验 | 降低 LLM 不稳定性对用户结果的影响 |
| Agent 工作台 | 在转换或打磨过程中展示计划、工具调用和异常提示 | 提升高级用户和演示场景下的可信度 |
| MCP/Skill 扩展 | 将资源、提示词、工具和领域规则标准化暴露或沉淀 | 让项目具备可复用、可集成的 Agent 能力 |

## 6. 非功能需求

| 类型 | 要求 |
|---|---|
| 性能 | 3 章小说（约 1-2 万字）转换时间 < 2 分钟 |
| 可用性 | 界面简洁，非专业用户无需学习即可使用 |
| 兼容性 | 支持 Chrome/Edge/Firefox 主流浏览器 |
| 可部署性 | 单机可运行，无需复杂基础设施 |
