## 1. 产品定位

**一句话价值主张：** Lively Novel 让不懂剧本的小说作者，把自己的几章小说，一键变成**结构化 YAML 剧本**——格式正确、拆好场、对白清晰、内心戏已视觉化、人物/场景全局一致，可在 YAML 结构上逐场打磨，最终导出符合 Schema 的剧本文件。

**核心交付物（题目要求）：**
- **YAML 格式结构化剧本**（主交付物）
- **YAML Schema 文档 + 设计理由**（题目明确要求）

**差异化亮点：**

1. **内心戏"视觉化"转译** —— 把心理描写/独白转成动作、对白、画外音(V.O.)。最体现改编"灵魂"，最能体现 AI 价值。
2. **跨章全局一致性** —— 自动抽人物表/场景表/线索，保证 3+ 章连贯。体现"成体系"。
3. **打磨 = 编辑 YAML** —— 生成后可直接编辑结构化字段，或让 AI 按新手法重生成。体现"可进一步打磨"。
4. **Agent 化演进** —— 在 `master` 分支逐步把固定 LLM workflow 改造为可观测、可恢复、可受控工具调用的 Agent 工作台。

## 2. 用户流程

### 2.1 端到端旅程

**小林写了一部 5 章网文，想改成短剧剧本。她打开 Lively Novel——**

#### ① 导入小说

她上传 5 章小说的 `.txt` 文件。系统自动识别章节边界。

- **MVP**：上传 `.txt`
- **扩展 TODO**：`.epub`、`.md`、`.docx`、`.pdf` → 统一先转成 `.md`（AI 最易理解的格式），再进入转换管线

#### ② 一键转换（两阶段）

她选择剧本类型（影视/短剧/广播剧/话剧等），然后点"开始转换"。

**关键设计：转换分两阶段，不是一次性生成。**

**阶段 A · 滚动全局状态**（融入逐场转换）
- 当前实现不再额外前置通读全文，而是在场景生成后持续更新"滚动全局状态"
- 提取：截至当前场景的剧情概要、承上启下上下文、近期活跃人物、近期事件线、人物列表及关系、故事线事件、重复意象/物件、内部时间线、伏笔状态
- 产出：一份随场景推进持续累积的"全局认知"，最终填入 YAML 顶层 `plotSummary / characters / storylines`
- 内部状态：`contextSummary / activeCharacters / activeThreads / motifs / timeline / foreshadows` 只用于后续场景上下文，不进入最终 YAML
- 前端展示：逐场生成进度 + 全局状态更新提示

**阶段 B · 逐场生成**（带着全局认知，从头逐场写剧本）
- 基于已累积的全局认知，逐场生成结构化剧本
- 前端可流式展示每场生成过程
- 不同剧本类型影响：场景密度、对白/动作比例、转场方式、是否用画外音等

**剧本类型选择**（转换前让作者选）：
- 动画剧本（TV单集，~20-24min）← MVP 默认
- 短剧剧本（~2-5min/集）
- 广播剧剧本
- 话剧剧本
- ……（可扩展）

#### ③ 预览剧本

转换完成，她看到一份完整的剧本——
- **左侧**：剧本渲染视图（标准剧本排版：场景标题居中加粗、角色名大写、对白居中、动作行左对齐）
- **右侧**：结构化大纲（场景列表 + 人物出场一览）

#### ④ 全局信息可视化（加分亮点）

- **人物关系图谱**（节点=角色，边=关系，可交互）
- **剧情主线/支线事件节点**（时间线视图）
- **人物间互动随时间变化**（热力图/曲线）
- **"世界地图" + 人物/事件随时间演进**（地理+时间双轴）

> 这些可视化是**加分项**，不是 MVP 必做。但如果有时间做 1-2 个，demo 视频里会非常出彩。

#### ⑤ 逐场打磨（编辑 YAML 结构）

她发现第 7 场的内心戏转译不够好——原文"她心里一阵酸楚"被转成了旁白，但她想改成动作。她点击那一场，进入**YAML 编辑模式**：
- 左侧直接编辑本场 YAML 结构（heading / scriptBlocks / sourceChapter）
- 右侧实时渲染 PREVIEW，按剧本阅读顺序展示 SHOT / ACTION / INSERT / SFX / DIALOGUE / VO / TRANSITION
- 点击"保存"后把本场 YAML 变更持久化，最终进入导出链路
- 点击"取消"只丢弃当前未保存修改，回到最近一次保存后的草稿

> 当前 MVP 聚焦"手动编辑 YAML + 右侧实时预览 + 保存后导出"这一闭环。选择内心戏视觉化手法、整场 AI 重生、风格提示重写属于后续扩展，不作为当前已完成能力描述。

#### ⑥ 导出

- **YAML 文件**（核心交付）—— 符合 Schema 定义的结构化剧本，可在任意编辑器继续打磨
- 导出页展示带行号的 YAML 内容，支持复制与下载 `screenplay.yaml`
- 可读剧本（衍生）—— YAML 的人类友好渲染视图，用于阅读/分享，当前作为后续扩展

#### ⑦ Agent 工作台（后续演进）

当前页面链路仍以稳定的导入、转换、预览、打磨、导出为主。Agent 化后，转换页和打磨页会逐步增加可观测的 Agent 运行信息：

- Agent 生成的计划或固定计划。
- 每一步调用的工具，例如读取章节、切分场景、生成单场、校验 YAML、保存场景。
- Guardrail 检查结果，例如语言漂移、schema 错误、人物名不一致。
- 失败恢复建议，例如继续转换、重点打磨某场、重新生成某个片段。

这些信息用于提升开发、演示和高级用户信任感，不应把普通用户界面变成调试日志。默认视图仍保持简洁，Agent trace 可以作为可折叠面板或开发视图呈现。

## 3. MVP 范围定义

| 必做（MVP） | 加分亮点（时间允许就做） |
|---|---|
| ① 上传 txt + 历史小说复用 | ① .epub/.md/.docx/.pdf 支持 |
| ② 两阶段转换 + SSE 流式进度 | ② AI 单场重生 / 视觉化手法切换 |
| ③ 剧本预览渲染 | ③ 可读剧本导出 |
| ④ 逐场打磨（编辑 YAML） | ④ 可视化（人物图谱/时间线/地图） |
| ⑤ YAML 导出（核心） | |
| **YAML Schema 文档 + 设计理由**（题目要求） | |

### 3.1 Agent 化范围

| 阶段 | 产品目标 | 用户可见变化 |
|---|---|---|
| Phase 1 | 固定计划 Agent + trace | 转换过程可看到更清晰的步骤和工具调用状态 |
| Phase 2 | 受限 Planner | 系统能解释本次转换计划，失败时可回退固定计划 |
| Phase 3 | Critic 与自修复 | 异常场景可自动检查并给出修复或打磨提示 |
| Phase 4 | 多 Agent | 生成、连续性检查、打磨由不同专家能力协同完成 |
| Phase 5 | MCP | 外部 Agent 客户端可受控调用 Lively Novel 能力 |
| Phase 6 | Skill | 动画剧本、连续性、YAML 打磨规则沉淀为可复用能力包 |

## 4. 页面结构

### 4.1 页面列表

| 页面 | 功能 | MVP |
|---|---|---|
| 导入页 | 上传小说文本，选择剧本类型 | ✅ |
| 转换中页 | 展示逐场生成进度 + 全局状态更新提示 | ✅ |
| 剧本预览页 | 左侧剧本渲染 + 右侧大纲 | ✅ |
| 场景编辑页 | 单场 YAML 编辑、保存、取消、右侧实时预览 | ✅ |
| 全局信息页 | 人物表/场景表/可视化 | 加分 |
| 导出页 | 展示、复制、下载最终 YAML | ✅ |

### 4.2 核心交互说明

- **导入页 → 转换中页**：点击"开始转换"后自动跳转，无需额外操作
- **转换中页 → 剧本预览页**：转换完成后自动跳转
- **剧本预览页 ↔ 场景编辑页**：点击某场进入编辑，保存后更新后端持久化结果
- **剧本预览页 → 导出页**：点击"导出"按钮
- **场景编辑页 → 导出页**：不提供单独导出入口，导出统一留在"导出"页

## 5. 交付顺序（阶段性纵向切片，master 始终可跑）

| 阶段 | 内容 | 产出 PR |
|---|---|---|
| 0 | 骨架：`/frontend` + `/backend` 跑通空链路 | 1-2 PR |
| 1 | Schema 先行：定义剧本 YAML Schema + 设计理由文档 | 1-2 PR |
| 2 | 最小转换切片：粘贴一段 → 后端调 DeepSeek → 返回单场结构化 YAML → 前端展示 | 2-3 PR |

| 4 | 内心戏视觉化（差异化①） | 1-2 PR |
| 5 | 跨章一致性：章节分割 + 全局人物/场景/线索表（差异化②） | 2-3 PR |
| 6 | 可读剧本预览渲染 | 1-2 PR |
| 7 | 人机打磨闭环：逐场重生/改风格/编辑/再导出（差异化③） | 2-3 PR |
| 8 | 打磨 + README + demo 视频 | 1-2 PR |
