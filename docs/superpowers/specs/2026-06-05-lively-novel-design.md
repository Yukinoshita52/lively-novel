# Lively Novel — 产品设计文档

> 日期：2026-06-05
> 状态：定稿（待用户确认后进入架构设计）

---

## 1. 项目背景

Lively Novel 是**七牛云 72h 项目开发**比赛的参赛作品，选题为**题目三：AI 小说转剧本工具**。

项目名含义：小说是一种**静态文本载体**，而剧本是一种**面向表演和视觉呈现的动态载体**。Lively Novel 帮助作者将静态小说转化为能够"动起来"的剧本初稿。

---

## 2. 目标客户

### 主客户：原创/网文作者

**画像：** 写了小说、梦想 IP 改编（影视/短剧/广播剧/漫画）的人。他们有故事，但**完全不懂剧本格式**。

**为什么选他们：**
- 题目原文"降低改编门槛"最贴合这类人群
- 他们是痛点最深的人——有创作力但缺专业知识和转换能力
- 最容易做出有共鸣的产品故事和 demo

### 次要受益者

| 人群 | 特征 | 受益方式 |
|---|---|---|
| 新手/改编编剧 | 懂剧本格式，要效率 | 省掉机械拆场、提白、排版 |
| 短剧/广播剧团队 | 要批量、快、标准化 | 批量转换、一致性保证 |
| 影视公司 IP 评估 | 快速判断小说可改编性 | 快速看到"改出来长什么样" |

## 3. 用户痛点（7 个，全部要解决）

按真实发生顺序排列：

| # | 痛点 | 描述 | 严重度 |
|---|---|---|---|
| 1 | 格式是黑箱 | 剧本格式（场景标题、动作行、角色提示、对白、括号提示、转场）对小说作者是天书 | ★★★★ |
| 2 | 对白埋在散文里 | "'你够了。'她别过脸，声音发颤"——要把"谁、说了什么、带着什么情绪"干净地抽出来 | ★★★★ |
| 3 | 内心戏无处安放 | 小说大量靠心理描写和内心独白。镜头拍不出"想法"，必须转成动作/对白/画外音(V.O.)/可视细节。**这是改编的灵魂，也是作者最容易卡死的地方** | ★★★★★ |
| 4 | 不知道在哪"拆场" | 小说一章是连续叙述，剧本要按地点/时间切成一个个场景 | ★★★★ |
| 5 | "讲述"要变"呈现" | 小说能直接写"她很绝望"，剧本得靠看得见的动作演出来 | ★★★★ |
| 6 | 跨章一致性 | 3 章以上，人物名、地点、线索得统一，需要一份全局的人物表/场景表 | ★★★ |
| 7 | 纯体力活劝退 | 就算硬做，机械重排和标注也耗到放弃 | ★★★★ |

### 用户的真实诉求（Job-to-be-done）

> 「我有几章小说，我想**不用学剧本知识、不做体力活**，就快速拿到一份**格式正确、结构清晰、能看出戏剧雏形**的剧本初稿，然后我自己在上面改。」

### 核心矛盾

**小说是"静态叙述"，剧本是"动态呈现"，中间这层转译，作者既缺知识（不懂格式）、又缺能力（不会把内心戏视觉化）。** Lively Novel 要做的就是替他跨过这层。

## 4. 产品形态

### 4.1 端到端使用流程

**小林写了一部 5 章网文，想改成短剧剧本。她打开 Lively Novel——**

#### ① 导入小说

她把 5 章文本粘贴进输入区（或上传 `.txt` 文件）。系统自动识别章节边界。

- **MVP**：粘贴文本 / 上传 `.txt`
- **扩展 TODO**：`.epub`、`.md`、`.docx`、`.pdf` → 统一先转成 `.md`（AI 最易理解的格式），再进入转换管线

#### ② 一键转换（两阶段）

她选择剧本类型（影视/短剧/广播剧/话剧等），然后点"开始转换"。

**关键设计：转换分两阶段，不是一次性生成。**

**阶段 A · 全局分析**（先跑一遍全文）
- 提取：大致剧情走向、人物列表及关系、场景列表、关键事件、时间线
- 产出：一份"全局认知"（供阶段 B 参考，保证跨章一致性）
- 前端展示：分析进度 + 分析结果摘要

**阶段 B · 逐场生成**（带着全局认知，从头逐场写剧本）
- 基于阶段 A 的全局认知，逐场生成结构化剧本
- 前端可流式展示每场生成过程
- 不同剧本类型影响：场景密度、对白/动作比例、转场方式、是否用画外音等

**剧本类型选择**（转换前让作者选）：
- 影视剧本（长片，~90-120min）
- 短剧剧本（~2-5min/集）
- 广播剧剧本
- 话剧剧本
- ……（可扩展）

#### ③ 预览剧本

转换完成，她看到一份完整的剧本——
- **左侧**：剧本渲染视图（标准剧本排版：场景标题居中加粗、角色名大写、对白居中、动作行左对齐）
- **右侧**：结构化大纲（场景列表 + 人物出场一览）

#### ④ 全局信息可视化（加分亮点）

- **人物关系图谱**（节点=角色，边=关系，可交互）
- **剧情主线/支线事件节点**（时间线视图）
- **人物间互动随时间变化**（热力图/曲线）
- **"世界地图" + 人物/事件随时间演进**（地理+时间双轴）

> 这些可视化是**加分项**，不是 MVP 必做。但如果有时间做 1-2 个，demo 视频里会非常出彩。

#### ⑤ 逐场打磨

她发现第 7 场的内心戏转译不够好——原文"她心里一阵酸楚"被转成了旁白，但她想改成动作。她点击那一场，进入**编辑模式**：
- 手动改文本
- "重新生成本场"让 AI 换一种视觉化方式
- 选风格提示（如"更含蓄""更戏剧化"）
- 修改后实时更新渲染

#### ⑥ 导出

- 导出 YAML（结构化剧本，符合我们定义的 Schema）
- 导出可读剧本（排版文本）

| ③ 剧本预览渲染 | ④ 可视化（人物图谱/时间线/地图） |
| ⑤ 逐场打磨 | |

## 6. 差异化亮点

在"地基"（拆场/提白/标准格式）之上，三个重点打透的差异化方向：

1. **内心戏"视觉化"转译** —— 把心理描写/独白转成动作、对白、画外音(V.O.)。最体现改编"灵魂"，最能体现 AI 价值。
2. **跨章全局一致性** —— 自动抽人物表/场景表/线索，保证 3+ 章连贯。体现"成体系"。
3. **可编辑打磨闭环** —— 生成后能逐场重生/调整风格，人机协作打磨。体现"可进一步打磨"。

## 7. 交付顺序（纵向切片，main 始终可跑）

| 阶段 | 内容 | 产出 PR |
|---|---|---|
| 0 | 骨架：`/frontend` + `/backend` 跑通空链路 | 1-2 PR |
| 1 | Schema 先行：定义剧本 YAML Schema + 设计理由文档 | 1-2 PR |
| 2 | 最小转换切片：粘贴一段 → 后端调 DeepSeek → 返回单场结构化 YAML → 前端展示 | 2-3 PR |
| 3 | 结构化地基：拆场 + 提白 + 标准字段（多场景） | 2-3 PR |
| 4 | 内心戏视觉化（差异化①） | 1-2 PR |
| 5 | 跨章一致性：章节分割 + 全局人物/场景/线索表（差异化②） | 2-3 PR |
| 6 | 可读剧本预览渲染 | 1-2 PR |
| 7 | 人机打磨闭环：逐场重生/改风格/编辑/再导出（差异化③） | 2-3 PR |
| 8 | 打磨 + README + demo 视频 | 1-2 PR |

## 8. 技术约束

- **后端**：Java（Spring Boot）
- **前端**：Node（React 或 Vue）
- **大模型**：DeepSeek-v4（Pro / flash），OpenAI 兼容接口
- **输出格式**：YAML（结构化剧本）+ 可读文本
- **额外交付物**：YAML Schema 设计文档（含设计理由）
- **比赛时间窗口**：2026-06-05 00:00 → 2026-06-08 00:00（72h）
